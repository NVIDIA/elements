# Slots

Slots enable dynamic content to be rendered into the template of a given element.

📘 Tip: general rule, if the text will be visible to the user then likely the API should use a Slot.

📘 Tip: slot projection of text content, enables apps to use i18n strategy of their choice


## Flexibility

Slots provide flexibility for content that is controlled by the host application and cannot be
guaranteed to be represented accurately through a specific element API.

```html
<mlv-alert status="success">
  <p>Hello <strong>There</strong>!</p>
</mlv-alert>
```

## Composition

Slots enable an easy way for component composition and decoupling from precise use cases that
are application specific. This also decouples layout behaviors such as icon and button placement,
giving the application control on DOM ordering.

```html
<mlv-card>
  <mlv-card-header>
    <h3>Plugin</h3>
    <a href="#">Learn More</a>
  </mlv-card-header>
  <mlv-card-body>
    <p>Cool new plugin features!</p>
  </mlv-card-body>
  <mlv-card-footer>
    <mlv-button>Enable</mlv-button>
  </mlv-card-footer>
</mlv-card>
```

Avoid exposing slot APIs as part of the public API. Rather than requiring the consumer to set a
slot name on elements, have the element assign its own slot automatically.

```typescript
class Modal {
  redner() {
  	return `
    <slot name="header"></slot>
    <slot></slot>
    <slot name="footer"></slot>
    `;
  }
}

class ModalFooter {
  connectedCallback() {
    this.slot = this.slot ?? 'footer';
  }
}
```

If the footer element assigns its own slot it prevents the consumer from having to know or remember
the underlying slot names reducing the API surface area.

## Rendering

Composition based APIs also provide flexibility for the host application to choose the most appropriate
render strategy for large lists of elements. For example, a tree view component can have hundreds or thousands
of potential nodes of varying complexity.

```html
<mlv-tree>
  <mlv-tree-item>item</mlv-tree-item>
  <mlv-tree-item>item</mlv-tree-item>
  <mlv-tree-item>item</mlv-tree-item>
</mlv-tree>
```

```html
<mlv-tree>
  <mlv-tree-item>item</mlv-tree-item>
  <mlv-tree-item>item</mlv-tree-item>
  <mlv-tree-item expanded>
  	<mlv-tree-item>item</mlv-tree-item>
    <mlv-tree-item>item</mlv-tree-item>
    <mlv-tree-item>item</mlv-tree-item>
  </mlv-tree-node>
</mlv-tree>
```

By leveraging composition the host application has full render control in their native framework API.

```html
<!-- Angular -->
<mlv-tree>
  <mlv-tree-item *ngFor="let item of items">{{item.id}}</mlv-tree-item>
</mlv-tree>
```

```tsx
// JSX
<mlv-tree>
  {items.map(item => <mlv-tree-item>{item.id}</mlv-tree-item>)}
</mlv-tree>
```

This enables full control of conditional rendering of tree nodes as well as more advanced features like lazy and virtual scrolling.

```html
<!-- Angular, only create nodes if the item meets a certain condition -->
<mlv-tree>
  <ng-container *ngFor="let item of items">
    <mlv-tree-item *ngIf="item.active">{{item.id}}</mlv-tree-item>
  </ng-container>
</mlv-tree>
```

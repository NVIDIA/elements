# Properties & Attributes

Properties/attributes represent an element's visual state. Examples:

- _types_ `status: error | success | warning`
- _states_ `expanded | selected | disabled`
- _behaviors_ `closable | draggable`

```html
<mlv-alert status="success"></mlv-alert>
```

Properties (`@property`) in lit enables the property to be set on the element via JavaScript or HTML Attributes.

```html
<mlv-alert status="success"></mlv-alert>

<script type="module">
  const alert = document.querySelector('mlv-alert');
  alert.status = 'warning';
</script>
```

## Impossible states

Properties/Attributes should avoid enabling “impossible states”. Impossible states are caused by conflicting APIs on a single element.

<mlv-alert status="success">Do:</mlv-alert>

```html
<mlv-alert status="success">hello there</mlv-alert>
```

<mlv-alert status="danger">Don't:</mlv-alert>

```html
<mlv-alert info>hello there</mlv-alert>
<mlv-alert success>hello there</mlv-alert>
<mlv-alert warning>hello there</mlv-alert>
<mlv-alert danger>hello there</mlv-alert>

<!-- alert cannot be in a success and error state -->
<mlv-alert success danger>hello there</mlv-alert>
```

<mlv-alert><mlv-icon slot="icon">🎓</mlv-icon> Learn:&nbsp;<a href="https://kentcdodds.com/blog/make-impossible-states-impossible" mlv-text="link">Make impossible states impossible</a></mlv-alert>

## Boolean Types

Boolean property/attribute type behave the same as native HTML element boolean types. The presence of the attribute is truthy and absence of the attribute is falsy. ([MDN Reference](https://developer.mozilla.org/en-US/docs/Web/HTML/Global_attributes/hidden))

```html
<mlv-alert closable>hello there</mlv-alert>
```

<mlv-alert status="warning"><p mlv-text="body">Warning: avoid double negations on attributes as this breaks the default expectations of native HTML elements, examples (<code mlv-text="code">disable-closable</code>, <code mlv-text="code">not-closable</code>, <code mlv-text="code">closable=”false”</code>)</p></mlv-alert>
&nbsp;
<mlv-alert status="warning"><p mlv-text="body">Warning: a custom coercion function is needed for react support which will set boolean attributes to a value of “false” <a href="https://github.com/vmware-clarity/core/blob/main/projects/core/src/internal/utils/dom.ts#L268" mlv-text="link">example</a> <a href="https://github.com/facebook/react/issues/9230" mlv-text="link">react issue</a>.</p></mlv-alert>

## Primitive Types

<mlv-alert status="accent">Note: Primitive type guidance is for atomic and highly reusable element libraries</mlv-alert>

While Lit keeps both properties and attributes in sync, do not use complex types like `object` and `array` on the API. Since HTML cannot represent JavaScript objects, Lit must JSON parse attributes and reflect them anytime there is a change to ensure the JavaScript property and HTML attribute are in sync. This can be expensive to parse when using an object or array and cause unexpected behaviors such as lost object references for the user.

<mlv-alert status="success"><p mlv-text="body">Use <code>string | number | boolean</code></p></mlv-alert>
&nbsp;
<mlv-alert status="danger"><p mlv-text="body">Avoid <code>object | array</code></p></mlv-alert>

Complex types cause compatibility and usability issues as it can require the developer to use JavaScript to render content. This can make it difficult for user-generated content like CMS systems or SSR (Server Side Rendering) to easily render static HTML.

<mlv-alert status="warning"><p mlv-text="body">Warning: avoid using <code>@property</code> on built in properties/attributes on the <code>HTMLElement</code> as the decorator overrides the getter/setter and can cause unexpected behavior.&nbsp;<a href="https://github.com/vmware-clarity/core/blob/main/projects/core/build/eslint-rules/reserved-property-names.js" mlv-text="link">read more</a></p></mlv-alert>
